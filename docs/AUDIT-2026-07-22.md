# Sweep — 2026-07-22

Static sweep of the repo plus a live check against the deployed backend and the
production database. Every claim below was verified, not inferred; the evidence
is inline. Nothing here has been fixed — this is the diagnosis.

Method: mobile↔backend contract diff against the live OpenAPI spec, nav-graph
extraction, live Supabase schema query, and booting the app in a headless
browser at phone size.

---

## 1. The date a client picks when posting a job is thrown away

**Severity: high — it sits in the middle of the core booking loop.**

`PostJobScreen` collects the preferred date as **free text** — a plain
`TextField` with placeholder `"Jul 5"` (`mobile/src/screens/client/PostJobScreen.js:461`).
Only the *time* gets a real picker. The value is then run through
`derivePreferredDate()` (`:63`), which is `new Date(rawString)`:

| Typed | Sent to the backend |
|---|---|
| `Jul 5` — **the app's own placeholder** | `2001-07-05` |
| `5 Jul`, `07/05` | `2001-07-05` |
| `Saturday`, `tomorrow`, `July 5th`, `asap` | dropped silently |
| `2026-07-25` | correct |

Bare month-day strings parse to **year 2001** in V8. Anything conversational
fails to parse and the field is omitted with no warning.

It does not stop there. `backend/app/api/interests.py:355` copies
`preferred_date` into `bookings.confirmed_date` when a quote is accepted, so a
wrong or missing value propagates into the booking.

Confirmed in production:

```sql
select count(*), count(preferred_date) from service_posts;
-- 35 rows, 0 with a preferred date
```

Zero of 35 posts have ever carried one — including live posts from today.

`@react-native-community/datetimepicker` is already a dependency and is already
used for the time field ten lines away, and again in `ConfirmDateCard.js`.

## 2. `service_posts` has two date columns, one of them a typo

Live schema has both **`preffered_date`** and **`preferred_date`**. Nothing in
the codebase references the misspelled one — `grep -rn preffered_date` across
`backend/`, `mobile/`, `docs/`, `web/` returns nothing. It is a dead column
shipped to production. Both are empty.

---

## 3. Backend features with no way to reach them

24 routes in the deployed API are never called by the app. Excluding health
checks, admin tooling and the Stripe webhook, these are shipped features with
no user-facing surface:

| Route | Note |
|---|---|
| `GET /me/credits` | credit ledger — no screen reads it |
| `POST /me/ghost`, `POST /me/unghost` | ghost mode — no screen reaches it |
| `POST /auth/refresh` | **never called; sessions just expire** |
| `POST /auth/logout` | never called |
| `POST /uploads/image` | while Profile still shows "Photo upload coming soon" |
| `GET /bookings/{id}/invoice.pdf` | no download entry point |
| `PATCH /disputes/{id}/resolve` | no resolution UI |
| `GET /reviews/client/{id}` | unused |
| `GET /service-posts/{id}` | unused (list endpoints are used) |

`/auth/refresh` is the one with user-visible consequences: the token is never
refreshed, so every session dies on expiry and the user is bounced to login.

## 4. The French and Arabic translations are mostly decorative

`mobile/src/i18n.js` defines 191 keys across EN/FR/AR. Only **103** are used.
Meanwhile **201 user-facing strings are hardcoded English** inside `<Text>`,
including `"Post a Job"`, `"Send quote"`, `"Something went wrong"`,
`"BEST VALUE"`, `"How did they do?"`, `"Verified"`.

Switching the app to French leaves most of it in English. Either finish the
extraction or drop the language selector before beta — a half-translated app
reads worse than an English-only one.

---

## 5. `CLAUDE.md` misdescribes the database

The master context file documents **12 tables**; the live database has **19**.

Undocumented: `disputes`, `referrals`, `user_credits`, `audit_log`,
`push_tokens`, `stripe_events`, `business_work_index`. `disputes` is missing
even though the dispute flow is a shipped screen.

It also documents `users.name`. The real columns are `first_name` / `last_name`.

Since every agent session reads this file first, wrong entries here propagate
into wrong code.

## 6. Dead and stubbed controls

Verified by hand (an automated pass produced mostly false positives on
multi-line JSX):

- **`ActiveBookingScreen.js:428`** — the phone button, labelled `"Call"`, is
  `onPress={() => {}}`. It is styled as live and does nothing.
- **`EarningsScreen.js:191`** — CSV export is `Alert.alert('Coming soon', …)`.
- **`EarningsScreen.js:110`** — the earnings chart is a placeholder.
- **`LoginScreen.js:122,128`** — "Continue with Apple" and "Continue with
  Google" are `disabled` at `opacity: 0.5`, and there are no OAuth routes in the
  backend. They read as greyed out, which is honest, but tapping gives no
  feedback and nothing says why. On the first screen a new user sees.
- **`PaymentMethodScreen`** — a "COMING SOON" Stripe stub.

`BookingDetailsScreen.js:764` also matched the detector but is correct — an
empty handler on a modal sheet to stop backdrop taps passing through.

## 9. Local environment is broken (not a code defect)

`mobile/node_modules` is **owned by root** — installed with `sudo` on Jul 15 —
and is missing two declared dependencies, `expo-local-authentication` and
`expo-updates`. Expo cannot resolve its config plugins through it, so
`npx expo start` fails outright on the Linux box.

```bash
sudo chown -R "$USER":"$USER" mobile/node_modules
cd mobile && npm install
```

An Android emulator is not viable on this machine regardless: 3.2 GB RAM total
with ~1.3 GB free, no JDK, no Android SDK, and the user is not in the `kvm`
group. Use the web build (see `tools/walkthrough/`) for flow and layout work,
and a real phone for anything native.

## 10. Smaller things

- **CORS blocks the web build.** `backend/app/main.py:73` allows only
  `localhost:5173` and `localhost:3000` plus `SWINGBY_ALLOWED_ORIGINS`. Any
  other origin cannot call the API. `tools/walkthrough/` sidesteps this with a
  same-origin proxy rather than widening the allowlist.
- **The offline banner misfires on web.** `OfflineBanner.js:21` requires both
  `isConnected === false` and `isInternetReachable === false`; NetInfo reports
  unreliably under React Native Web and the banner showed while the network was
  fine.
- **Onboarding wastes over half the screen.** At 390×844 the gap between the
  subtitle and the pagination dots is roughly 55% of the viewport.
- **69 hardcoded colour literals** across 33 files outside `theme/`, mostly
  `rgba(46,189,133,0.14)`-style status tints that should be tokens.

---

## Suggested order

1. The date field (#1) — it is in the booking loop and it is silently wrong.
2. `/auth/refresh` (#3) — users are being logged out for no reason.
3. `CLAUDE.md` (#5) — wrong context keeps producing wrong code.
4. The dead Call button (#6).
5. Decide on i18n (#4): finish it or hide the selector.
