- **Root cause (confirmed):** `backend/app/api/subscriptions.py:106` passes `STRIPE_PRICE_SOLO` / `STRIPE_PRICE_TEAM` env vars straight into `line_items[0].price`. Value in Render dev env is a **product** ID (`prod_...`), not a **price** ID (`price_...`). Stripe requires `price_...` here.
- **Fix:** Stripe dashboard → Products → find `prod_Un0szQtOA35l65` → copy its Price ID (`price_...`) → replace `STRIPE_PRICE_SOLO` (and re-check `STRIPE_PRICE_TEAM`) in dev env. Kira does the env-var swap; agent only verifies the code doesn't need change.

### A2 · UUID "threads" — APIError (prod, 125 hits, ongoing)
- **Symptom:** Postgres: `invalid input syntax for type uuid: "threads"`.
- **Confirmed frontend calls:** `mobile/src/screens/messages/MessagesScreen.js:162` and `mobile/src/screens/business/DashboardScreen.js:111` both `GET /messages/threads`.
- **Current code (correct on main):** `backend/app/api/messages.py:276` defines `/threads` BEFORE `/{booking_id}` (L460). This order should route `/messages/threads` to `list_threads`, not `get_messages`.
- **Hypothesis:** Render is running a stale commit where `/{booking_id}` was declared first — so `booking_id = "threads"` → `.eq("id", "threads")` → UUID error.
- **Steps:**
  1. Compare Render's deployed commit against `git log backend/app/api/messages.py` on `main`. If stale → redeploy.
  2. If deploy is current → add `@router.get("/threads/")` alias to catch trailing-slash calls, and check for older mobile builds still hitting a legacy path.
  3. Wrap `get_messages` with an explicit UUID validation → 404 instead of 500 so this class of error never reaches Postgres.
- **Verify:** hit `GET /messages/threads` with a valid JWT on Render; watch Sentry for 30 min post-deploy.
