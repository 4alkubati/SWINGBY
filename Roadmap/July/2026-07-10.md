The changes I’d make, in order:
Consolidate web/pre-launch and web/launch into one customer website.
This is the biggest structural win. They contain overlapping pages, components, auth, styling, and dependencies. Keep the deployed pre-launch site stable for now, migrate its useful pages into web/launch, then redirect it to the canonical site. Two customer-facing React apps will continuously drift.

Move SQL from docs/*.sql into supabase/migrations/.
Files such as [booking_events_and_photos.sql](/C:/Users/amrba/OneDrive/Desktop/AMR/10-SWINGBY/Swingby/docs/booking_events_and_photos.sql) are important database changes, not documentation. Numbered migrations give you ordering, repeatability, reviewability, and safer deployment. Keep schema explanations in docs/, but make migrations executable source of truth.

Add shared API contracts before extracting shared UI.
The bookings response mismatch I found is exactly what a contract package prevents. Start with schemas/types for responses such as BookingListResponse, payment states, and roles. The API validates them; web and mobile consume generated types or matching schemas.

Add a root workspace command layer.
Keep the deployment units separate, but add one root-level command interface:
npm run verify       # web lint/build + mobile checks
npm run test:e2e     # booking journey
npm run dev:web
npm run dev:mobile
npm run dev:api
Python can remain separately managed, but the root README should make the complete release check one clear operation.

Make documentation single-purpose.
Keep:
docs/ for engineering, operations, and legal implementation
planning/ for roadmap/daily work logs
marketing/ for go-to-market assets
design/ for visual assets and decisions
Your [WORKSPACE-MAP.md](/C:/Users/amrba/OneDrive/Desktop/AMR/10-SWINGBY/Swingby/WORKSPACE-MAP.md) already moves in this direction. I’d rename Roadmap/ to planning/ only after beta, to avoid breaking Obsidian links now.

Keep AGENTS/ at the repository root, but trim it.
It is a valid operational system, not product code. Keep only the current instructions, status, and active plan easily reachable; archive historical briefs and session logs so every future coding task does not have to navigate accumulated agent history.

What I would explicitly not do yet:
Do not move backend, mobile, or workers immediately; deploy configuration likely relies on them.
Do not build a large shared component library across web and React Native. Share tokens and API contracts first; their UI implementations should stay native to each platform.
Do not reorganize marketing, roadmap, or design files while payments and beta flows are still active.
The first practical milestone is: one customer website, migration-based database changes, and contract tests around the booking/payment APIs. That gives you the most reliability with the least churn.
