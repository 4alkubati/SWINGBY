# 🁢 D2.4 — Business subscription (off-platform monetization)

> Index: [[../DOMINOES|DOMINOES]] · Prev: [[D2.3-offplatform-pay|D2.3]] · Next: [[D2.5-status-cleanup|D2.5 — Status cleanup]] · Master log: [[_LEARNING-LOG]]

## 🎯 Goal

Every business has a subscription. Stripe handles billing. The app surfaces plan + status. Access gates on `subscription_status='active'`.

The 10% escrow cut on in-app card payments **stays as-is**. Subscription monetizes the off-platform flow + the "I want to be on the platform" right.

## 🤔 Why this matters

Off-platform bookings = no per-transaction cut. Without subscription, SwingBy earns $0 from cash/e-transfer customers (which will be most of them). Subscription is the bridge.

## ✅ NEEDS-KIRA — answered 2026-06-27

> Locked decisions. Code can proceed.

1. **Two-stream monetization model:**
   - **Customer side** — payment methods at booking: in-app Stripe escrow, Apple Pay, Google Pay, Cash. SwingBy takes a **10% platform cut from the customer total** on every in-app card/Apple/Google transaction (not deducted from the business — the business sees gross). Cash never touches SwingBy.
   - **Business side** — flat monthly **membership** (this domino). The membership monetizes off-platform bookings + the right to be on the platform at all.
2. **Membership tiers (CAD/month):**
   - **Solo** — $30/mo
   - **Teams** (any business with ≥1 employee) — $80/mo flat
   - **Enterprise** — contact sales (custom pricing, talk to Kira)
3. **Gate:** Business **cannot accept a booking** unless `subscription_status ∈ ('active','trialing')`. They can still browse the marketplace, edit profile, receive messages, and quote on posts. The 402 fires at the "Accept" tap — highest-intent moment.
4. **Beta posture (default — Kira override if needed):** Track-only during beta (every business auto = `trialing` indefinitely, no Stripe charge). Stripe billing flips on at public launch (Phase 3). This matches the "don't scare testers" stance. **If you want live Stripe sandbox billing during beta instead, say so and we'll wire the Checkout Session.**

## 🪜 Step-by-step

### 1. Schema migration — businesses subscription fields
```sql
ALTER TABLE businesses
  ADD COLUMN IF NOT EXISTS subscription_tier text
    CHECK (subscription_tier IN ('solo','team','enterprise')) DEFAULT 'solo',
  ADD COLUMN IF NOT EXISTS subscription_status text
    CHECK (subscription_status IN ('trialing','active','past_due','canceled','incomplete')) DEFAULT 'trialing',
  ADD COLUMN IF NOT EXISTS subscription_id text,                    -- Stripe sub_xxx
  ADD COLUMN IF NOT EXISTS stripe_customer_id text,                 -- cus_xxx
  ADD COLUMN IF NOT EXISTS subscription_price_id text,              -- price_xxx (which Stripe price they're on)
  ADD COLUMN IF NOT EXISTS subscription_started_at timestamptz,
  ADD COLUMN IF NOT EXISTS subscription_current_period_end timestamptz,
  ADD COLUMN IF NOT EXISTS subscription_cancel_at timestamptz;
```
`subscription_tier` auto-derived from `employees` count on every accept-employee trigger: 0 employees → `solo` ($30); ≥1 → `team` ($80); `enterprise` set manually by SwingBy team.

### 2. Stripe products + prices (TWO recurring prices)
- Dashboard (test mode) → Products → Add product → "SwingBy Business — Solo" → recurring $30 CAD/mo. Copy `price_xxx` → env `STRIPE_PRICE_SOLO`.
- Add product → "SwingBy Business — Team" → recurring $80 CAD/mo. Copy `price_xxx` → env `STRIPE_PRICE_TEAM`.
- Enterprise tier has no Stripe price (custom invoice handled by Kira out-of-band).

### 3. Backend — subscribe endpoint
File: `[[../../backend/app/api/subscriptions.py]]` (new).

```
POST /businesses/me/subscribe
auth: business owner
behavior:
  - resolve target tier from current employees count (solo if 0, team if ≥1)
  - pick STRIPE_PRICE_SOLO or STRIPE_PRICE_TEAM accordingly
  - if businesses.stripe_customer_id is null: create Stripe customer with email + name
  - create Stripe Checkout Session in 'subscription' mode with line_items=[{price, quantity:1}]
  - return checkout URL
```

```
GET /businesses/me/subscription
→ {status, current_period_end, cancel_at, manage_url}
manage_url: create a Stripe Billing Portal session, return URL
```

### 4. Backend — webhook handler (extend [[../../backend/app/api/payments_stripe.py|payments_stripe.py]])
Handle:
- `customer.subscription.created`
- `customer.subscription.updated`
- `customer.subscription.deleted`
- `invoice.payment_succeeded`
- `invoice.payment_failed`

For each: update `businesses.subscription_status`, `subscription_current_period_end`, `subscription_cancel_at` based on Stripe payload.

### 5. Backend — gate booking acceptance
File: `[[../../backend/app/api/bookings.py]]` — wherever the business accepts an interest / creates a booking.

```python
business = ...
if business.subscription_status not in ('active','trialing'):
    raise HTTPException(402, "Subscription required to accept bookings")
```

(Or whatever the user picked in NEEDS-KIRA #3.)

### 6. Mobile — new "Plan" section on [[../../mobile/src/screens/business/BusinessProfileScreen|BusinessProfileScreen]]
Card showing:
- Current plan + price.
- Status badge (Active / Trialing / Past Due / Canceled).
- Next billing date.
- **"Manage subscription"** button → `Linking.openURL(manage_url)` (opens Stripe Billing Portal).
- If status is `incomplete` or new business: **"Start subscription"** primary CTA → POST `/subscribe` → open returned Checkout URL.

### 7. Mobile — onboarding hook on [[../../mobile/src/screens/onboarding/BusinessSetupScreen|BusinessSetupScreen]]
At end of business setup, push to a small "Start your subscription" screen (reuse Plan card). Beta posture (NEEDS-KIRA #1) drives whether this is required or skippable.

### 8. Mobile — failure UX when gating fires
When accept-interest fails with 402:
- Toast: "Subscription required to accept bookings."
- Auto-navigate to BusinessProfile → Plan section, scroll to it.

### 9. Test the full loop
