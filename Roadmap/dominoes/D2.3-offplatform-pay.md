# 🁢 D2.3 — Off-platform "mark as paid" (cash + e-transfer)

> Index: [[../DOMINOES|DOMINOES]] · Prev: [[D2.2-invoices|D2.2]] · Next: [[D2.4-business-subscription|D2.4 — Business subscription]] · Master log: [[_LEARNING-LOG]]

## 🎯 Goal

A client can mark a booking paid via **cash** or **e-transfer**, recorded by SwingBy without SwingBy ever touching the money. Both sides see "Paid (cash)" on the booking and receipt. **No automatic platform cut** on these — monetization comes via [[D2.4-business-subscription|D2.4]].

## 🤔 Why this matters

Tradespeople take cash/e-transfer constantly. If SwingBy *requires* card payment, the business loses bookings and goes back to texting. Recording off-platform payments keeps the **trust layer + receipt + Live Job Status** signal intact even when money flows outside the app.

## ✅ Pre-reqs

- [ ] [[D2.2-invoices|D2.2]] done so the receipt screen has a `payment.method` field to render.
- [ ] Supabase MCP access.

## 🪜 Step-by-step

### 1. Schema migration — extend `payments.status` enum
Migration name: `add_payment_status_off_platform`.

```sql
ALTER TABLE payments
  DROP CONSTRAINT IF EXISTS payments_status_check,
  ADD CONSTRAINT payments_status_check
    CHECK (status IN ('pending','paid_full','paid_off_platform','refunded','failed'));

ALTER TABLE payments
  ADD COLUMN IF NOT EXISTS method text
    CHECK (method IN ('stripe_card','cash','e_transfer','other'));
```

Apply via Supabase MCP `apply_migration`.
**Verify:** `\d payments` shows the new constraint + column.

### 2. Backend — new endpoint
File: `[[../../backend/app/api/payments_offplatform.py]]` (new).

```
POST /bookings/{booking_id}/mark-paid-offplatform
body: {method: "cash" | "e_transfer" | "other", note?: string}
auth: client only (`bookings.client_id == current_user`)
behavior:
  - 400 if booking.status != 'completed'
  - 400 if a payment row already exists with status in (paid_full, paid_off_platform)
  - INSERT payments {booking_id, total_charged=booking.total_amount, escrow_held=0, released_to_business=0, platform_cut=0, status='paid_off_platform', method=<method>}
  - UPDATE bookings SET payment_status='fully_released' WHERE id=<id>  -- end state for off-platform
  - INSERT booking_events {event_type='paid_offplatform', note=method+optional}
  - return new payments row
```

**Verify:** curl test against a completed test booking.

### 3. Backend — register in main.py
`app.include_router(payments_offplatform.router, tags=["payments"])`.

### 4. Mobile — UI on [[../../mobile/src/screens/client/BookingDetailsScreen|BookingDetailsScreen]]
- When `booking.status === 'completed'` AND no `payment` exists OR `payment.status === 'pending'`:
  - Show **two** primary buttons side-by-side: **"Pay with card"** (existing) + **"Mark as paid (cash/e-transfer)"** (new).
- Tapping the new button → modal asking method (Cash / E-Transfer / Other) → confirm → POST → refresh.
- After success: pill flips to "Paid (cash)" — uses the same `paymentPillLabel` map (extend it).

### 5. Mobile — extend `paymentPillLabel`
Add cases for `paid_off_platform` → "PAID OFF-PLATFORM". Color: `colors.accent` (distinguish from green "PAID").

### 6. Business-side visibility
- [[../../mobile/src/screens/business/JobManagementScreen|JobManagementScreen]] Status tab → when off-platform paid event exists in the timeline, render a chip "Marked as paid by client via {method}".

### 7. Receipt integration
[[../../mobile/src/screens/shared/InvoiceScreen|InvoiceScreen]] already renders `payment.method` from [[D2.2-invoices|D2.2]] — confirm it now shows "Cash" or "E-transfer" correctly for off-platform.

### 8. Manual test
- Client completes a booking, marks it paid via cash → both sides see correct state on booking + receipt. No money moved. No platform_cut recorded.
